## 工作纪律

- **澄清优先**：当用户处于澄清意图、讨论设计方案、审阅文档阶段时，不可以直接改动代码，只可以根据指示改动相关文档。
- **确认后执行**：只有用户明确指示"开干"、"写"、"执行"等可执行指令时，才开始写代码。

## 重构笔记（source/git → source/version 的启发）

### 命名的演变就是认知的深化

- `source/git.rs` → `source/git_tag.rs` → `source/version.rs`
- 每次改名都暴露了上一层的误解。模块的价值不是"用了 git"，而是"把 git tag 作为版本号的事实源"。
- 命名问自己：这个模块的**调用方用它来做什么**，而不是**它内部用了什么技术**。

### Trait 的价值不是多态，是划定测试边界

- 即使只有 1 个生产实现，trait + mock 也能解锁单元测试覆盖：纯 @[test] 从 5 个增长到 25 个。
- 不是所有抽象都需要运行时多态——编译期绑定 trait + mock 就足够。
- 多出来的测试覆盖了 scope 过滤规则、兜底逻辑、semver 排序、一致性判断——全是容易出 bug 且有业务价值的地方。没有 trait 就测不到。

### 抽 I/O 边界是提升测试覆盖率最高效的动作

- 把 `read_config_versions` 从 `contract::version` 移到 `source::version` 后，单元测试从 5 个 → 35 个。
- 不是因为写了更多代码，而是因为把 I/O 逻辑放到了本来就是放 I/O 逻辑的地方。
- 重构的核心是找到"这里应该能测但测不到"的那条分界线。
